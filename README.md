# 마이리얼트립 실시간 정책 통합 조회 플러그인

## 1. 무엇을, 누가, 어떤 상황에서 쓰나요?

마이리얼트립 CS 상담원, 또는 문의를 넣는 고객 본인이 씁니다. 항공권 일정변경,
투어 취소, 예약 오류처럼 **지금 당장 답이 필요한 문의**가 대상입니다. 문의를
넣으면 상품 유형(항공권/투어·액티비티/숙소)별로 흩어져 있는 공개 정책·운영시간·
수수료 정보를 실시간으로 훑어서, "확인 후 안내드리겠습니다"로 시간을 끄는 대신
지금 가능한 옵션과 예상 처리 시간을 바로 내놓습니다.

## 2. 왜 이 문제를 선택했나요?

처음엔 "CS 자동 응대 봇"으로 접근했는데, 곧 의문이 들었습니다. 이 정도는 회사가
이미 내부에서 충분히 만들 수 있는 수준 아닌가. 그런데 안 만들었다는 건 이유가
있을 거라고 봤고, 실제 후기를 뒤져보니 답이 나왔습니다.

- 항공 일정변경 문의는 1:1 채팅에 안내만 있고 구체적 날짜 선택 화면이 없으며,
  처리에 1~3영업일이 걸린다는 후기가 확인됩니다.
- 전화 상담은 09:00~18:00(항공권 환불/변경은 평일 09:00~17:00)로 운영시간이
  제한적이고, 파트너 지원센터도 평일 09:00~18:00만 운영됩니다.
- 예약이 확정으로 표시됐지만 실제로는 공급사 사정으로 불가능했던 사례(스노클링
  투어 전체 미운영 통보)에서도, 대체 옵션을 즉시 안내받지 못하고 연락이 끊기는
  경험이 반복적으로 보고됩니다.

즉 채널이 없어서가 아니라, **채널은 있는데 실시간 통합 답변이 안 나온다**는
게 진짜 문제였습니다. 항공사/현지 공급사/숙소마다 정책과 재고 데이터가 제각각
흩어져 있어서, 이걸 실시간으로 합치는 자체가 구조적으로 어렵다는 뜻입니다.
그래서 "CS 챗봇" 대신 "파편화된 정책을 통합 조회하는 문제"로 다시 좁혔습니다.
이전에 팀으로 여행 서비스를 만들면서 지도·대중교통·장소 정보 API처럼 서로 다른
외부 데이터를 하나의 판단으로 묶는 과정에서 겪었던 어려움과 결이 같은 문제라,
그 경험을 살려 이 스킬을 설계했습니다.

## 3. 플러그인은 어떻게 작동하나요?

`src/skills/realtime-policy-resolver/SKILL.md`에 정의한 절차 그대로 움직입니다.

1. **문의 분류**: 상품 유형, 요청 유형(일정변경/취소/오류), 긴급도(출발까지 남은 시간)를 먼저 가려냅니다.
2. **공개 정책 조회**: 마이리얼트립 고객센터 페이지, 항공사 공개 변경 규정, 공급사 공지, 숙소 취소 정책 등 공개 자료만 봅니다. 내부 시스템 접근은 애초에 가정하지 않습니다.
3. **옵션 계산**: 조회한 정책을 조합해서 지금 가능한 옵션, 예상 처리 시간, 확인 안 되는 항목을 갈라 산출합니다.
4. **정형 출력**: 문의 요약 / 지금 가능한 옵션 / 예상 처리 시간 / 상담원 확인 필요 항목 / 참고 출처 순서로 출력합니다.

모든 항목에 출처 URL을 반드시 붙이고, 공개 자료로 확인 안 되는 건 "확인 불가"로
못 박습니다. 지어내지 않습니다.

## 4. AI를 어떻게 활용했나요?

- Codex로 (1) `SKILL.md`에 정의한 4단계 절차(문의 분류 → 공개 정책 조회 →
  옵션 계산 → 정형 출력)를 짜고, (2) 실제 플러그인(`mrt-realtime-policy-resolver`)으로
  패키징한 뒤, (3) `docs/test-cases.md`의 샘플 문의 3건을 돌려서 검증했습니다.
- 스킬 안에 "추측 금지 / 출처 필수 / 확인 불가 항목 분리"라는 원칙을 명시적으로
  박아서, Codex가 공개 자료로 뒷받침 안 되는 내용을 확정적으로 말하지 못하게
  막았습니다. 이건 이전에 팀으로 여행 서비스를 개발하면서 데이터 검증에 써온
  원칙("검증된 수치만 쓰고, 출처 불명 정보는 근거로 안 쳐준다")을 그대로 옮긴 겁니다.
- 실행해보니 Codex가 실제로 마이리얼트립 공개 정책 문서와 상품 페이지를 조회해서
  옵션과 처리 시간을 뽑아냈고, 항공사별 수수료처럼 입력 정보가 부족한 부분은
  알아서 "확인 불가"로 분리해 답했습니다. 실행 로그는
  `logs/codex-execution-log-20260710.md`와 `logs/codex/*.jsonl`(원본)에 그대로
  남아 있습니다.

## 5. 어떻게 검증했나요?

- 문제를 정의할 때 쓴 근거(CS 운영시간, 처리 소요시간, 이용자 후기)는 전부 공개
  URL로 확인됩니다. 아래 출처 목록에 그대로 남겨뒀습니다.
- 스킬 자체에 "확인 불가 항목을 명시적으로 분리"하는 원칙을 넣어서, 검증 안 된
  내용을 확정적으로 던지지 않게 만들었습니다.
- (실행 검증) Codex 앱에 플러그인(`mrt-realtime-policy-resolver@personal`)을
  설치하고 `docs/test-cases.md`의 샘플 문의 3건(항공권 일정변경 / 투어 취소·대체 /
  예약 오류)을 실제로 돌렸습니다. 세 케이스 다 SKILL.md에 정의한 5개 섹션 형식을
  지켰고, 옵션·처리시간에 출처가 붙었으며, 공개 자료로 확인 안 되는 항목(항공사별
  수수료, 익일 실시간 재고 등)은 지어내지 않고 "확인 불가"로 갈라놨습니다. 실행
  결과 전문은 `logs/codex-execution-log-20260710.md`(요약)와
  `logs/codex/*.jsonl`(원본)에 그대로 남겨뒀습니다.

## logs 폴더 안내

- `logs/codex/*.jsonl` — **원본 로그**입니다. `log-hooks.zip` 훅을 통해 Codex 세션 종료 시 자동으로 저장된, 가공되지 않은 실제 대화 기록입니다.
- `logs/codex-execution-log-20260710.md` — 위 원본 로그 중 케이스 1~3 실행 결과를 심사자가 빠르게 훑어볼 수 있도록 정리한 **참고용 요약**입니다. 원본과 다른 내용은 없으며, 검증의 최종 근거는 `logs/codex/*.jsonl` 원본입니다.

## 설치 및 실행 방법 (Codex CLI 기준)

1. `src` 폴더를 로컬 플러그인 경로로 복사하거나, 프로젝트 루트에 `.agents/plugins/marketplace.json`을 추가해 `source.path`가 `src`를 가리키도록 설정합니다.
2. Codex를 재시작한 뒤 `/plugins`에서 `myrealtrip-realtime-policy-resolver`가 보이는지 확인합니다.
3. 프롬프트에서 예시 문의(예: "내일 세부 출발인데 스노클링 투어가 갑자기 이번달 전체 운영 안 한다고 해요")를 입력해 스킬이 트리거되는지 확인합니다.
